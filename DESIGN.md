# Fudaba(札場) 設計

2026-07-14制定。人間とAIが共有する作業アイテム(札)の台帳+カンバンビュー。Jimboの住民(独立リポジトリ+submodule)であり、開発管理プラットフォーム構想(Jimbo/ARCHITECTURE.md が正典)のPhase Aを担う。

## 何であるか

- 人間はUIで、AIはMCPで、**同じ札を読み書きする**。札はサーバー側SQLiteに永続し、AIセッションの消失に巻き込まれない
- WBS・バグシート・タスクリストの最小共通形を単一のドメイン型「札」で表す(種別で判別)
- 名前の由来: 村の高札場。札が掛かり、皆が見に来る場所

## ドメイン型

```
札 {
  id: 札ID(採番),
  種別: '実装' | 'バグ' | '新仕様' | '仕様検討' | 'タスク分解' | '記録' | '決定',
  タイトル: string,
  本文: string,
  状態: '未着手' | '進行中' | '完了' | 'ブロック',
  担当者: string | 未割当,       // メンバー名。人間もAIも同格
  作成者: string,
  リンク: { ルーム名?: string } , // 弱参照(ID参照のみ。外部キーは張らない)
  作成時刻 / 更新時刻
}
```

- 状態遷移に制約は設けない(どの状態からどの状態へも変更可。運用の柔軟性優先)
- 種別は「その文章を次にどう扱うか」を表す。実装・バグは実行、仕様検討・タスク分解は整理、新仕様・記録・決定は知識の状態を区別する
- 「決定」「記録」も状態を持つ(決定=完了で作るのが自然だが強制しない)
- 旧データの「タスク」は「実装」、「メモ」は「記録」へ起動時に冪等移行する

## 提供する3点セット(プラグイン型住民の標準形)

1. **サーバールート登録関数**: `Fudabaルートを登録する(app: FastifyInstance, ストア)` — 自前ポートなし。ワークスペースサーバー(AgentRoomサーバー :7100)に間借り
   - REST: GET/POST `/api/fudaba/items`、PATCH `/api/fudaba/items/:id`
   - ストレージ: 専用SQLiteファイル(fudaba.sqlite3、パスは注入)。AgentRoomのDBとは別ファイル
2. **UIビュー部品**: 状態4列のカンバン(SengenUI + Vanilla Extract、テーマはホストのCSS変数を継承、スマホ狭幅対応)。ホストはAgentRoom UIシェル(=ワークスペースシェル。LAN配信でスマホからも見える)
3. **MCPツール登録関数**: `fudaba_create` / `fudaba_update` / `fudaba_list`(+help文書)。Jimbo MCP(:7110)に相乗り

配線はホスト側(Jimbo main.ts / AgentRoom UIシェル)のコンポジションルートが行う。Fudabaは配線先を知らない。

## dogfooding

Phase B以降(エージェント稼働可視化・成果物共有・BoomYackビュー統合)のタスクは、完成したFudaba自身に札として登録して管理する。

## ラベル(2026-07-14追加、ユーザー要求)

- 札は**0個以上**のラベル(自由文字列タグ)を持つ。未分類の札を許すのが本質(GitHub issueの「書く前にリポジトリを決めさせられる」面倒さの否定)
- ラベルは事前登録不要。既存札のラベル集合を候補表示(担当者datalistと同パターン)
- リポジトリ名(jimbo / fudaba / boomyack / agentroom等)は特別な概念ではなくラベルの慣用
- ビューはラベル・種別・担当者・状態のフィルタを合成できる(カンバン上部、クリックで絞り込み)
- MCP: fudaba_create/updateにラベル配列、fudaba_listにラベルフィルタ。運用ガイドに「所属が決まらないアイデアはラベルなしで投げてよい。分解するAIが適切なラベルを付ける」を明記

## 添付(2026-07-14追加、ユーザー要求)

- 札は0個以上の画像添付(png/jpg/jpeg/gif/webp、1件10MB上限)を持てる。動機はバグ調査中のスクリーンショット共有
- 添付作成は札作成APIと分離する。UIは先に札を作成し、返されたIDへ保留中の添付を順次送信する。追加・削除は専用エンドポイントを使う(`POST /api/fudaba/items/:id/attachments`、`GET /api/fudaba/attachments/:保存名`、`DELETE /api/fudaba/items/:id/attachments/:保存名`)
- バイナリ本体はDBに置かずファイルシステム(添付ストレージ、パスはホストが注入)へ保存し、DBのattachments列にはメタ情報(保存名・元ファイル名・バイト数・追加時刻)のJSON配列だけを持つ(DB肥大化回避)
- 保存名はサーバーが乱数から採番する(32桁16進数+拡張子)。GET/DELETEの保存名パラメータはこの形式へ厳密一致させ、パストラバーサルを構造的に排除する(ホワイトリスト方式)
- UI: 作成時と更新時の両方でファイル選択・ドラッグ&ドロップ・本文欄への画像貼り付けを受け付ける。本文欄は内容に追従して自動伸長し、Ctrl/Cmd+Enterで保存する。詳細ではサムネイル一覧・原寸表示オーバーレイ・削除も提供する
- MCP: fudaba_listの応答（札DTO）に添付メタ一覧が自然と含まれる。AIが「スクショあり」を認知できるが、画像バイナリ自体を取得するAI向けツールは未実装(次の札候補)

## UX憲法: ページ遷移ゼロ原則(2026-07-14制定)

GitHub issueが使われなかった元凶はページ遷移の多さ(ユーザーの実体験)。Fudabaのすべての操作(フィルタ・ラベル編集・詳細閲覧編集・作成)は**同一画面内**(カンバン+サイドパネル)で完結させる。新しいビューを追加するときも、この原則を破る設計(詳細専用ページ・編集専用ページ)は採らない。

## 右側UIの責務境界(2026-07-16制定)

- Jimbo全体の右サイドバーは、プロジェクトファイルなどワークスペース全体に属する機能を置く
- AgentRoomの右サイドバーは、ルーム移動・参加者・稼働状態など会話コンテキストに属する機能を置く
- Fudabaの札詳細はFudabaビュー内の右インスペクターレールに置く。札の選択・編集はFudaba自身の責務であり、iframe境界を越えてホストへ注入しない
- 狭幅画面では同じ詳細部品を全画面シートとして表示する。配置だけを変え、札編集の責務は分けない
